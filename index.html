Project 01
FitCheck — AI Virtual Try-On
A project that started from a real personal question: how would a trench coat actually look on someone with proportions like mine, not on a very tall model? The process led me through failed 3D attempts, MediaPipe experimentation, and a pivot toward generative image workflows.
Gemini API MediaPipe Image generation Prompt design Workflow pivoting
What problem did I encounter?

I wanted to know how a trench coat would look on someone with my proportions, not on a tall model in an online store. My first instinct was to build a real virtual try-on experience that could save a trip to a physical store. But I quickly discovered that realistic try-on is much harder than it looks, especially when product information from the store is limited.

What did I build / experiment with?

I first tried a 3D-oriented approach using MediaPipe. The core issue was trying to place a 2D garment onto a 3D body representation, which did not look realistic or convincing. That experiment showed me the limitations of the approach: missing garment geometry, limited store data, and a mismatch between how the clothing asset existed and how I wanted to represent the person.

How did I solve it?

Instead of forcing a weak solution, I pivoted. I changed the concept from strict virtual try-on into FitCheck: a workflow where the user provides a garment URL and selects parameters such as height, skin tone, and body-related attributes, and the system generates an image with the Gemini API to approximate how the item may look on someone with those characteristics. This was a much stronger creative AI solution because it used generation where simulation was failing.

What I learned

This project taught me that experimentation matters more than forcing the original idea. I learned when to abandon a technically elegant direction that was not producing good outputs, and instead design a workflow that better fits the available data and current model capabilities. It also gave me hands-on experience combining prompt design, parameter-based generation, model experimentation, and product reframing.

Project 02
Focus Study Platform
A study tool designed during wartime to help students recover concentration after interruptions such as sirens. This project combined timer logic, user flow design, and prompt engineering for empathy and focus recovery.
React Gemini API Prompt engineering UX thinking Workflow design
What problem did I encounter?

This project started from a problem I personally felt and also heard from people around me. During the war, semester B started at my university, and students needed to keep studying while living with sudden sirens. The hard part was not only the interruption itself, but the fact that after returning from the shelter, it was difficult to get back into the same mental focus.

What did I build / experiment with?

I built a platform where users can set study time, break time, and task goals, then begin a focus session. The system includes a button the user can press if a siren happens. When that happens, a Gemini-powered LLM flow begins: it tells the user to go to a protected space, waits there, and when the user returns, it responds empathetically and asks what they were working on immediately before the interruption.

How did I solve it?

I wrote prompts that explicitly emphasized that the model should be empathetic, calming, and grounding. I wanted the assistant not just to respond correctly, but to speak in a way that helps the user feel supported and gradually return to focus. I also considered connecting the system to the Home Front Command API so the timer would stop automatically, but I intentionally chose a manual button-based workflow because pressing the button gives the user a sense of control in a situation defined by lack of control.

What I learned

This project taught me that with LLMs, you can shape not only what the model does but also the emotional tone it brings into the user experience. I learned how prompt design can influence reassurance, trust, and attention recovery, and how workflow design choices can be just as important as technical automation.

Project 03
Echo — AI-moderated chat
A chat platform for discussing controversial topics, where I explored a core creative AI challenge: how to define, detect, and rephrase offensive language in a way that is consistent, useful, and fair.
Groq API Prompt engineering Temperature tuning PostgreSQL Model experimentation
What problem did I encounter?

While building Echo, I ran into a surprisingly hard question: what actually counts as offensive? At first, my prompt simply asked the model whether a message was offensive or disrespectful, and if so, to suggest three better alternatives. But the results were unstable. Some clearly acceptable messages passed, while others did not. One especially surprising example was the simple message “hi”, which the model sometimes treated as offensive and sometimes did not.

What did I build / experiment with?

To understand the behavior, I experimented directly with the model and also used the Groq desktop chat to ask why “hi” could be interpreted as offensive. The explanation was that a very short message may be interpreted as rude, abrupt, or low-effort. I also tested temperature changes, but parameter tuning alone did not solve the inconsistency. That made it clear that the issue was not just randomness — it was prompt ambiguity.

How did I solve it?

I rewrote the moderation prompt into a much more detailed instruction set with clear acceptance and rejection rules, giving the model stronger boundaries around what should count as offensive and what should not. This improved consistency and made the system more aligned with the product goal. The process taught me that LLMs are not binary machines with clean 0/1 logic. They are probabilistic systems, and the quality of their behavior depends heavily on how precisely you frame the task.

What I learned

This project gave me hands-on experience with prompt design, model interpretation, workflow iteration, and parameter testing. More importantly, it taught me that building AI systems is not only about calling an API — it is about defining concepts clearly enough that the model can act on them consistently in a real product context.
